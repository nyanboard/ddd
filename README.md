二、分阶段详解
① 需求阶段：搞清楚"做什么"
主导：产品经理（PM） — 调研用户 / 市场，输出 PRD（需求文档）、原型草图。
并行：业务分析师（BA） — 梳理业务流程、规则、边界，把需求拆成功能点（ToB 项目尤其重要）。
并行：UX 用研 — 访谈、问卷，验证需求是否真实存在。
✅ 产出：PRD、需求清单、优先级

② 设计阶段：定"长什么样、好不好用"
主导：UX 设计师（先） — 信息架构、交互流程、低保真原型。
衔接：UI 设计师（后） — 高保真视觉稿、设计规范、切图标注。
并行：架构师 — 提前评估技术可行性，避免设计做出来做不了。
PM 把关 — 评审设计稿是否符合需求。
✅ 产出：设计稿、交互原型、设计规范

③ 技术方案阶段：定"怎么做"
主导：架构师 — 技术选型、系统架构图、数据库设计、接口约定。
并行：前端 TL / 后端 TL — 评审可行性、工作量评估、任务拆解。
并行：DevOps — 搭建开发环境、代码仓库、CI/CD 流水线骨架。
并行：QA — 阅读需求，提前开始写测试用例（不等到开发完）。
✅ 产出：技术方案文档、接口文档、排期表、测试用例初稿

④ 开发阶段：写代码（多人并行）
前端工程师 — 界面、交互、对接接口。
后端工程师 — 业务逻辑、API、数据库。
移动端工程师 — App / 小程序（如有）。
⬆️ 三者基本是并行开发，靠"接口文档"对齐。
DevOps 持续保障 — CI 自动构建、测试环境部署。
PM / QA 关键节点验收 — 按里程碑看 demo。
✅ 产出：可运行的代码、自测通过的版本

⑤ 测试阶段：找 bug
主导：测试工程师（QA） — 功能测试、回归测试、缺陷反馈。
并行：测试开发（SDET） — 跑自动化测试、性能 / 压力测试。
开发配合 — 修 bug、改问题（被动响应，非并行）。
✅ 产出：测试报告、可发布版本

⑥ 上线阶段：发布到生产
主导：DevOps — 执行部署、灰度发布、回滚预案。
安全工程师 — 上线前安全审核（漏洞、权限、敏感数据）。
PM 验收 — 确认线上功能符合预期。
架构师 / 技术负责人 — 应急指挥。
✅ 产出：线上可用系统

⑦ 运维与迭代：让它跑得稳、跑得好
SRE / 运维 — 监控、告警、故障处理。
安全工程师 — 持续巡检、应急响应。
数据分析师 — 看埋点数据、AB 实验。
运营 / 市场 — 推广、用户反馈收集。
PM — 收集反馈 → 开启下一轮需求（回到第 ① 步，形成闭环）。
三、AI + 飞书串联方案（含角色）
把飞书当作"协作 + 数据 + 触发"的中枢，AI 当作"智能引擎"注入到每个环节，让阶段之间、角色之间自动流转。

（一）核心架构：3 层
┌─────────────────────────────────────────────────────────┐
│  智能层  AI（飞书智能伙伴 / 外接大模型如 Claude、豆包）       │
│          → 写文档、做小结、生成用例、分类缺陷、根因初判        │
├─────────────────────────────────────────────────────────┤
│  触发层  飞书机器人 + Webhook + 多维表格"自动化"             │
│          → 阶段间自动流转、@通知、状态联动                   │
├─────────────────────────────────────────────────────────┤
│  协作层  飞书文档 / 群 / 会议(妙记) / 审批 / 飞书项目         │
│          → 人讨论、评审、签字的地方                         │
├─────────────────────────────────────────────────────────┤
│  数据层  飞书多维表格 = 全流程"数据底座"                    │
│          → 需求池 / 任务库 / 缺陷库 / 发布记录，一表贯穿      │
└─────────────────────────────────────────────────────────┘
一句话：数据落在多维表格 → 用文档和群协作 → 用机器人/Webhook 在阶段间自动触发 → 用 AI 在每个节点做增值。

（二）逐阶段：飞书载体 / AI / 触发方式
阶段	飞书载体	AI 干什么	触发 / 串联方式
①需求	妙记会议 + 文档 PRD + 需求池	妙记自动转录访谈、AI 生成会议小结；AI 辅助写 PRD 初稿	PRD 定稿 → 多维表格自动建行 → 群机器人 @设计
②设计	文档(设计稿内嵌) + 群评审	AI 对照 PRD 评审设计是否覆盖需求、生成设计说明	设计稿评审通过 → 表格状态 →"待排期" → @架构师
③技术方案	文档(技术方案/接口) + 飞书项目	AI 根据需求生成接口草案、拆任务、估工时	接口文档确定 → 自动拆任务入"任务库" → @开发
④开发	飞书项目 + 群 + Git(Webhook)	AI 编程助手(Cursor/Claude Code)写代码；机器人自动 Code Review 小结	代码提交 → Webhook → 机器人推群 + CI 结果回写表格
⑤测试	多维表格缺陷库 + 群	AI 从 PRD 批量生成测试用例；新缺陷 AI 自动分类/指派/定级	缺陷入表 → AI 字段填负责人 → @对应开发 → 修复后状态联动
⑥上线	飞书审批 + 群	AI 生成发布说明(changelog)	审批通过 → 触发部署 Webhook → 机器人广播"已上线"
⑦运维	监控 Webhook + 群 + 工单表	告警进群后 AI 先做根因初判 + 处置建议，并自动建工单	告警 → Webhook → AI 小结 → @值班人 → 数据回流闭环到 ①
（三）阶段间是怎么"自动串起来"的
①需求PRD定稿 ──webhook──▶ 多维表格新建需求行 ──@──▶ ②设计评审群
                                          │
②设计通过 ──表格状态变──▶ ③架构师收到通知 ──产出接口文档
                                          │
③接口定稿 ──AI拆任务──▶ 任务库 ──@──▶ ④开发群(前端/后端并行)
                                          │
④Git提交 ──webhook──▶ CI跑完 ──结果回群+表格 ──▶ ⑤测试
                                          │
⑤缺陷入表 ──AI指派──▶ @开发修 ──回归通过──▶ ⑥审批
                                          │
⑥审批通过 ──webhook──▶ 部署 ──机器人广播──▶ ⑦监控值守
                                          │
⑦线上反馈/数据 ──AI汇总──▶ 回流需求池 = 回到 ①（闭环）
每个"箭头"都是一次自动触发，不需要人去手动通知、手动建任务——这是"串行起来"的精髓。

各环节用什么工具 / 流程（建议，可替换）
思路：让每个环节的产出结构化、可被 AI 消费，下游（尤其开发）就能直接拿来用。

环节	主导角色	工具 / 流程（推荐 + 备选）	交接物
①需求	PM / BA / 用研	访谈：飞书妙记转录 + AI 小结；问卷：飞书问卷 / 问卷星；行为数据：神策 / Mixpanel；PRD 用 OpenSpec / Spec 格式写在飞书文档（结构化、AI 可直接消费）；需求池进多维表格	结构化 PRD / Spec
②设计	UX → UI	原型 + 视觉：Figma / 即时设计 / MasterGo；组件库沉淀为设计系统（Figma DS）；交付标注：蓝湖 / Figma Dev Mode	设计稿 + 标注
③技术方案	架构师 / TL	接口设计：Apifox / YApi / Swagger；架构图：Mermaid / Excalidraw / 飞书白板；OpenSpec 把 spec 拆成任务入飞书项目排期	接口文档 + 任务清单
④开发	前端 / 后端 / 移动端	OpenSpec 模式：spec → AI agent 规划 → AI 实现（Cursor / Claude Code / 通义灵码）→ 自动生成 PR；代码托管 GitLab / GitHub；Code Review 用机器人 + CodeRabbit；CI 用 GitLab CI / Jenkins；质量门禁 SonarQube + ESLint；特性开关（Unleash / Apollo）做灰度	可测试代码 + PR
⑤测试	QA / SDET	用例管理：多维表格 / TestRail；自动化：Playwright / Cypress（UI）、Pytest（接口）、k6（性能）；AI 从 spec 生成用例；缺陷入多维表格（自动 @ 开发）	测试报告 + 缺陷库
⑥上线	DevOps / 安全 / PM	审批：飞书审批流；部署：GitLab CI / ArgoCD + K8s；镜像扫描 Trivy；安全 SonarQube / Snyk；灰度 Unleash / Apollo；回滚 ArgoCD	线上版本
⑦运维	SRE / 数据 / 运营	监控：Prometheus + Grafana → 飞书 Webhook；日志 ELK / Loki；链路 SkyWalking；AB 实验 GrowthBook / 火山 DataTester；用户反馈飞书客服 → AI 汇总 → 回流需求池	数据洞察 + 反馈（回 ①）
💡 关键串联点：既然开发用 OpenSpec，让 PM 在 ① 阶段就直接用 OpenSpec / Spec 格式写 PRD——PRD 即 spec，一路流到开发被 AI 直接消费，省掉"PRD 再翻译成技术语言"这一步。需求与开发用同一种"语言"，是这套流程最大的提效点。

（六）真实串行示例：一个需求从提出到上线
全程没人手动 @ 过任何人：

PM 在文档写完 PRD → 多维表格自动建需求行 → 机器人 @UX/UI
UX/UI 设计稿评审通过 → 表格状态变 → 机器人 @架构师
架构师 写完接口文档 → AI 拆出 5 个任务入任务库 → 机器人 @前端TL/后端TL 排期 → @工程师 开工
工程师 提交代码 → Git Webhook → CI 跑完，结果回群 + 表格 → 机器人 @QA "可测试"
QA 测出 bug 入表 → AI 自动指派 → @对应工程师 修复 → 回归通过 → 表格状态→"待发布"
PM/安全 在审批流点"通过" → 触发部署 Webhook → DevOps 灰度上线 → 机器人广播"已上线"
数据/运营 看数据、收反馈 → AI 汇总 → 回流需求池 → @PM 开启下一轮（闭环）
（七）最小可用版本（先搭这 3 件套）
不用一上来全做，先跑通 MVP：

1 张多维表格 — 列出：需求 / 任务 / 缺陷 / 发布 4 个视图（一张表用"类型"字段区分即可）。
1 个机器人 — 接 Open Platform，能收发消息 + 接 Webhook（Git、CI、监控都往这里推）。
飞书妙记 + 智能伙伴 — 需求会议自动转录小结，零开发成本。
跑顺之后，再逐个加：AI 生成用例、AI 指派缺陷、审批触发部署、告警 AI 根因。

（三）状态流转（状态机）——串联的核心
状态的每一次变化 = 一次自动触发 = @下一个角色。 这是整个串联的引擎。

需求池：

草稿 → 评审中 → 待排期 → 设计中 → 开发中 → 测试中 → 已发布
                ↓                                    ↑
              已拒绝                              (闭环回①)
   (@PM评审)  (@设计)  (@架构师)  (@开发)  (@QA)  (@DevOps/广播)
任务库：

待开始 → 进行中 → 待评审 → 已完成
            ↓ ↑
         已阻塞
   (@工程师开工)      (@TL/Code Review)   (@QA可测试)
缺陷库：

新建 → 已指派 → 修复中 → 待验证 → 已关闭
                       ↑        ↓ (回归失败)
                       └─ 重新打开
   (AI自动指派@开发)          (@QA验证)     (@QA关闭)
发布记录：

待审批 → 审批中 → 灰度中 → 已发布
              ↓
           已回滚
   (@PM/安全审批)  (@DevOps部署)  (机器人广播"已上线")

（五）自动化规则（多维表格「自动化」配置示例）
触发条件（当…）	执行动作（则…）	串联到谁
需求状态 →「待排期」	发送消息到 #设计群	@UX/UI
需求状态 →「开发中」	发送消息到 #开发群	@工程师
任务状态 →「已完成」 且 该需求下所有任务都完成	需求状态自动 →「测试中」	@QA
缺陷新建	AI 字段生成分类 → 自动填「指派给」	@对应开发
缺陷状态 →「待验证」	发送消息到 #测试群	@QA 回归
发布状态 →「审批中」	触发飞书审批流	@PM/安全
发布状态 →「已发布」	发送消息到全员群	广播 changelog
第 3 条尤其关键：用汇总（rollup）判断"需求的所有任务是否完成"，自动把需求推进到测试阶段——这就是"代码写完自动流转给 QA"的实现。

（六）关联字段带来的「全链路可追溯」
从一张发布记录往回追，一键看到：

发布 V1.2.0
  ├─ 包含需求 RQ-0012「购物车改版」
  │     ├─ 任务 T-0034（前端 @张三 已完成）
  │     ├─ 任务 T-0035（后端 @李四 已完成）
  │     └─ 缺陷 BUG-0089（已关闭）
  ├─ 包含缺陷 BUG-0090（线上紧急修复）
  └─ 发布说明（AI 生成）
出问题能秒级定位到人；做复盘能看清一个版本动了哪些需求、修了哪些 bug。这就是"数据底座"的价值。
